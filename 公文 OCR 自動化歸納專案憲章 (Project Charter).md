# **公文 OCR 自動化歸納專案憲章 (Project Charter)**

**版本：** V1.0  
**指導原則：** 規格驅動開發 (Specification-Driven Development, SDD)  
**目標架構：** 模組化設計、地端隱私保護、雲端自動化同步

## **1\. 專案概述與業務目標 (Project Overview & Business Objectives)**

### **1.1 核心痛點**

目前累積之公文影像檔（紙本掃描或圖檔）皆依賴人工逐一閱讀、提取關鍵欄位並手動登錄，整體流程耗時且容易產生人為疏失，形成資訊孤島，無法與後續的自動化歸納系統無縫串接。

### **1.2 專案目標**

開發一套具備高度可維護性與擴展性的公文自動化處理系統。系統透過地端多模態大型語言模型（Local Multimodal LLM）對公文圖檔進行端到端語意理解，精準擷取關鍵欄位並將其序列化（Serialization）為結構化 JSON 資料，最終實現本地端預覽編輯與 Google Workspace 雲端自動化同步存儲。

### **1.3 開發治理與規範 (Development Governance)**

*   **套件與環境管理：** 全案採用 `uv` 作為唯一的 Python 專案管理工具，負責依賴管理、虛擬環境建置與執行。所有相依性必須定義於 `pyproject.toml`，嚴禁手動使用 `pip` 進行非受控安裝。
*   **規格同步一致性：** 當系統規格或 Data Schema 發生異動時，必須同步更新所有相關規劃文件（包含但不限於：開發計畫書、WBS 任務確認清單、系統架構說明書），確保全案資訊源頭唯一。
*   **專案索引導引：** 專案實作啟動後，須於根目錄建立統一的專案函式與結構說明文件（例如 `PROJECT_REFERENCE.md`），彙整並摘要系統內所有函式 API、核心資料結構與 Schema 規格，作為 SDD 開發流程的導引核心。

### **1.4 成功標準 (KPI)**

* **欄位識別準確度：** 核心欄位（如發文字號、主旨、發文日期等）綜合識別準確度以 **95%** 為目標。  
* **架構擴展性：** OCR/LLM 驅動層與業務邏輯層、UI 介面層完全解耦，未來更換底層 AI 模型或 OCR 引擎時，核心業務代碼無需重構。

## **2\. 專案發展藍圖與範疇劃分 (Project Roadmap & Scope)**

本專案採用敏捷開發與 MVP（最小可行性產品）策略，區分為兩大階段實施：

| 發展階段 | 核心範疇與介面形式 | 資料儲存與自動化機制 |
| :---- | :---- | :---- |
| **Phase 1: MVP 快速消積木工具** | CLI 命令列介面 \+ 內建原生 File Dialog 視窗。支援單張或批次公文圖檔選取。 | 調用本地 Ollama (Qwen) 進行解析。引入 90 天日期校驗機制，直接輸出結構化 JSON 或追加至本地 CSV 檔。 |
| **Phase 2: 完整桌面應用程式** | Python \+ QML (PySide6/PyQt6) 精美桌面 GUI。採雙欄對照 UI（左側欄位編輯、右側圖片即時預覽）。 | 整合 Google Workspace API。自動將掃描圖檔上傳至 Google Drive，並將確認後的 OCR 結構化數據自動寫入 Google Sheets。 |

## **3\. 輸出資料合約規格 (Data Schema Specification)**

不論處於何種開發階段，系統核心解析器（Parser）皆須遵循相同的資料輸出合約，確保前後期資料格式完全兼容：  
`{`  
    `"doc_date": "YYYY-MM-DD",     // 公文日期：須由民國紀年自動轉換為標準西元 ISO-8601 格式`  
    `"doc_category": "字別字串",    // 公文分類/文號字別（例如："中分署訓"）`  
    `"doc_number": "文號字串",      // 公文文號：純數字字串（例如："1152301262"）`  
    `"doc_from": "機關全銜",        // 發文機關（例如："勞動部勞動力發展署中彰投分署"）`  
    `"case_officer": "姓名",        // 承辦人/聯絡人姓名`  
    `"related_class": "班別/專案",  // 相關班別或特定專案名稱（具備關鍵內容提煉能力）`  
    `"key_points": ["標籤1", "標籤2"] // 關鍵字/摘要：採字串陣列儲存。寫入 CSV/Sheets 時自動以逗號串接`  
`}`

## **4\. 品質保證與防錯機制 (QA & Data Validation Gate)**

### **4.1 日期時間窗口驗證 (Time-Window Validation)**

* **核心邏輯：** 系統讀取 doc\_date 後，自動計算該日期與今日（Current Date）之絕對值天數差：$| \\text{doc\\\_date} \- \\text{current\\\_date} |$。  
* **容許閾值 (Threshold)：** 設定為 **90 天**。  
* **例外處理：** 若天數差超過 90 天，系統判定可能存在模型幻覺（如民國年誤轉西元年），CLI 階段將輸出 \[WARNING\] 提示，GUI 階段則於介面醒目標記「需人工覆核」，但不阻斷核心批次寫入流程。

### **4.2 測試驅動規劃**

* **黃金數據集 (Golden Dataset)：** 建立包含 30\~50 份不同發文機關、排版樣式的真實公文影像作為基準測試集。  
* **邊界條件測試 (Edge Cases)：** 納入包含低解析度、影像傾斜、大紅/藍章遮擋文字之極端圖檔，用以驗證 System Prompt 於 Temperature \= 0.0 下的輸出穩定度。
